# Atomic Variables & Reflection API — Interview Notes

---

## Key Concepts

### Atomic Variables
- **`java.util.concurrent.atomic`** — a package of classes that support lock-free, thread-safe operations on single variables using hardware-level **CAS (Compare-And-Swap)** instructions.
- **CAS** — an atomic CPU instruction: "if the current value equals the expected value, replace it with the new value; otherwise do nothing and return the current value." No lock is acquired.
- **`AtomicInteger` / `AtomicLong` / `AtomicBoolean`** — atomic wrappers for primitive types. Key methods: `get()`, `set()`, `getAndIncrement()`, `incrementAndGet()`, `compareAndSet(expected, update)`.
- **`AtomicReference<V>`** — atomic wrapper for an object reference; allows lock-free updates to a shared pointer.
- **`AtomicStampedReference<V>`** — pairs a reference with an integer **stamp** (version counter) to solve the **ABA problem**.
- **`LongAdder` / `LongAccumulator`** (Java 8+) — higher-throughput alternatives to `AtomicLong` under high contention; internally stripe the counter across multiple cells and sum them on read. Ideal for counters/statistics; not suitable when you need a consistent snapshot mid-update.
- **`VarHandle`** (Java 9+) — a typed reference to a variable with support for plain, opaque, `acquire`/`release`, and volatile access modes. Replaces `sun.misc.Unsafe` for low-level atomic access in library code.

### Reflection API
- **`java.lang.reflect`** — a package that lets you inspect and manipulate classes, methods, fields, and constructors **at runtime**, regardless of access modifiers.
- **`Class<T>`** — the entry point to reflection; represents a loaded class or interface. Three ways to obtain: `MyClass.class` (compile-time), `obj.getClass()` (runtime), `Class.forName("fully.qualified.Name")` (dynamic loading).
- **`Field`** — represents a class field; `field.get(obj)` / `field.set(obj, value)` read and write it. Call `setAccessible(true)` to bypass `private`.
- **`Method`** — represents a method; `method.invoke(obj, args...)` calls it at runtime.
- **`Constructor<T>`** — represents a constructor; `constructor.newInstance(args...)` creates an instance.
- **`setAccessible(true)`** — suppresses Java access checks; allows reading/writing `private` fields and invoking `private` methods. Requires appropriate module or security permissions.
- **`Proxy.newProxyInstance()`** — creates a dynamic proxy that implements a set of interfaces and routes all method calls through an `InvocationHandler`. Foundation of Spring AOP and many mocking frameworks.
- **Annotations at runtime** — only annotations with `@Retention(RetentionPolicy.RUNTIME)` are visible to reflection; `SOURCE` and `CLASS` annotations are not.

## How It Works (Internals / Mechanics)

### CAS Loop (Optimistic Locking)
```
Thread reads current value V
Thread computes new value N
CAS(V, N):
  if memory still holds V → write N atomically → SUCCESS
  else → memory was changed by another thread → RETRY (spin)
```
```java
// AtomicInteger.incrementAndGet() is effectively:
int current, next;
do {
    current = get();        // volatile read
    next = current + 1;
} while (!compareAndSet(current, next));  // CAS retry loop
return next;
```
This is **optimistic** — no blocking, but under extreme contention threads spin. That's why `LongAdder` outperforms `AtomicLong` at high contention.

### ABA Problem & `AtomicStampedReference`
```
Thread 1 reads A
Thread 2 changes A → B → A  (value is back to A)
Thread 1's CAS succeeds even though the value changed twice
```
`AtomicStampedReference` adds a version stamp that increments on every write, so `CAS(A, stamp=1, newVal, newStamp=2)` fails if another thread changed the value and the stamp no longer matches.

### LongAdder Internal Striping
```
Low contention:    [ base cell ]  ← single CAS, fast path
High contention:   [ base ] [ cell-0 ] [ cell-1 ] [ cell-2 ] ...
                      ↑ each thread updates its own cell (less CAS retry)
sum() = base + cell-0 + cell-1 + ...   ← summed on read
```

### Reflection Method Invocation Pipeline
```
Method.invoke(obj, args)
    → AccessCheck (skipped if setAccessible(true))
    → Argument boxing/unboxing
    → MethodAccessor (JNI stub or generated bytecode after 15 invocations)
    → Actual method execution
```
- First ~15 invocations use a slow JNI-based accessor.
- After the inflation threshold, the JVM generates a dedicated bytecode accessor — much faster but still slower than a direct call due to boxing and stack-frame overhead.

### Module System & Reflection (Java 9+)
```
module com.framework {
    opens com.framework.internal to com.library; // selective deep reflection
}
```
- Strong encapsulation by default: `setAccessible(true)` on a type in a non-opened package throws `InaccessibleObjectException`.
- `--add-opens` JVM flag opens a package at launch (common migration workaround).

## Common Pitfalls & Edge Cases

### Atomic Variables
- **`AtomicInteger` is not a drop-in for `int` in all contexts** — it does not override `equals()` by value; `new AtomicInteger(5).equals(new AtomicInteger(5))` is `false`. Use `.get()` to compare values.
- **`compareAndSet` is not a substitute for a transaction** — you can atomically update one variable, but updating two `AtomicInteger`s together is not atomic. Use `synchronized` or `StampedLock` for multi-variable invariants.
- **`LongAdder` is not consistent under concurrent reads** — `sum()` is not a snapshot; other threads may be incrementing cells simultaneously. Use `AtomicLong` when you need a precise point-in-time value.
- **Spin loops under contention** — CAS retry loops waste CPU cycles when many threads compete. Prefer `LongAdder` for high-contention counters or back off with `Thread.onSpinWait()` (Java 9+).
- **ABA problem is real in linked data structures** — if you use `AtomicReference` for a lock-free linked list node, a node removed and re-added looks identical to CAS. Use `AtomicStampedReference` or immutable nodes.

### Reflection
- **Performance** — reflective calls are significantly slower than direct calls: argument boxing, access checks, and no JIT inlining. Never use reflection in a tight inner loop.
- **`setAccessible(true)` breaks encapsulation** — it bypasses `private`, making internal implementation details part of your contract. Breaks silently if the target class is refactored.
- **`Class.forName()` uses the calling class's class loader by default** — in multi-classloader environments (OSGi, app servers), pass the correct `ClassLoader` explicitly: `Class.forName(name, true, targetClassLoader)`.
- **Checked exceptions become `InvocationTargetException`** — `Method.invoke()` wraps any exception thrown by the target method in `InvocationTargetException`; always call `.getCause()` to retrieve the real exception.
- **`getFields()` vs `getDeclaredFields()`** — `getFields()` returns all **public** fields including inherited ones. `getDeclaredFields()` returns all fields (including private) declared in **that class only** — not inherited ones. The same distinction applies to `getMethods()` vs `getDeclaredMethods()`.
- **Annotations need `RUNTIME` retention** — frameworks like Spring read annotations reflectively; if you define a custom annotation without `@Retention(RetentionPolicy.RUNTIME)`, it will be invisible at runtime.

## Interview Q&A

### Conceptual Questions

**Q: What is CAS and how do atomic classes use it?**
> **A:** CAS (Compare-And-Swap) is a single CPU instruction that atomically checks whether a memory location holds an expected value and, if so, replaces it with a new value — all without acquiring a lock. Java's `AtomicInteger`, `AtomicLong`, etc. expose CAS through `compareAndSet(expected, update)`. Internally they delegate to `sun.misc.Unsafe` (pre-Java 9) or `VarHandle` (Java 9+), which map directly to the `CMPXCHG` (or equivalent) hardware instruction.

**Q: What is the ABA problem and how does `AtomicStampedReference` solve it?**
> **A:** The ABA problem occurs when a thread reads value A, another thread changes A→B→A, and then the first thread's CAS succeeds because the value appears unchanged — even though the underlying state has changed twice. This is dangerous in lock-free data structures (e.g., a removed-and-re-added list node). `AtomicStampedReference` pairs the reference with an integer stamp (version counter). Every update increments the stamp, so even if the reference returns to A, the stamp no longer matches and the CAS correctly fails.

**Q: When would you use `LongAdder` over `AtomicLong`?**
> **A:** Use `LongAdder` when you have **many threads incrementing a counter frequently** (high write contention) and you only need the total occasionally. It internally stripes the counter across multiple cells, reducing CAS contention — each thread mostly updates its own cell. Under low contention, `AtomicLong` is equally fast and gives you a precise real-time value with `get()`. Use `AtomicLong` when you need an accurate, consistent snapshot at any moment (e.g., sequence number generation, compare-and-swap logic).

**Q: What are the three ways to obtain a `Class<?>` object and when would you use each?**
> **A:** (1) `MyClass.class` — compile-time literal; the safest option, works with generics, no exception thrown. (2) `obj.getClass()` — runtime type of an existing instance; returns the actual concrete type even when the variable is declared as a supertype. (3) `Class.forName("com.example.MyClass")` — dynamic loading by name at runtime; used when the class name is not known at compile time (plugin systems, config-driven factories). The third form can throw `ClassNotFoundException`.

**Q: What is the difference between `getFields()` and `getDeclaredFields()`?**
> **A:** `getFields()` returns all **public** fields of the class and all its superclasses/interfaces. `getDeclaredFields()` returns **all fields** (public, protected, package-private, private) declared directly in that class, but **excludes inherited fields**. To walk the full hierarchy including private inherited fields, you must call `getDeclaredFields()` on each class in the superclass chain manually.

**Q: What happens when a reflectively invoked method throws an exception?**
> **A:** `Method.invoke()` wraps it in an `InvocationTargetException`. The original exception is accessible via `invocationTargetException.getCause()`. Checked exceptions thrown by the target become unchecked from the caller's perspective because `invoke()` itself only declares `InvocationTargetException`. Forgetting to unwrap `.getCause()` is a very common bug — the stack trace points to the reflection machinery rather than the real failure site.

**Q: How does `setAccessible(true)` work and what are its risks?**
> **A:** `setAccessible(true)` on a `Field`, `Method`, or `Constructor` suppresses Java's access control checks for that reflective object. In Java 8 and below, any code could call it. From Java 9 onward, it requires that the target class's package is **opened** (via `module-info.java` or `--add-opens`); otherwise it throws `InaccessibleObjectException`. The risks: it breaks encapsulation (your code couples to private internals), fields can be mutated outside the class's own invariants, and it can be a security vulnerability if untrusted code reaches it.

**Q: What is a `VarHandle` and why was it introduced?**
> **A:** `VarHandle` (Java 9, `java.lang.invoke`) is a typed reference to a variable (field, array element, off-heap memory) that provides atomic and memory-ordered access modes: plain, opaque, acquire/release, and volatile. It was introduced to replace the internal `sun.misc.Unsafe` API (which is not part of the public spec) with a supported, type-safe alternative. Library authors use it to implement lock-free data structures with fine-grained memory ordering guarantees without incurring the overhead of full `volatile` semantics everywhere.

### Scenario-Based Questions

**Q: You have a counter incremented by 500 threads. The final value is consistently less than expected. What went wrong and how do you fix it?**
> **A:** The counter is being updated with a non-atomic read-modify-write, e.g. `count++` on a plain `int` or even a `volatile int`. Even `volatile` doesn't make `++` atomic. Fix options in order of preference for a pure counter: (1) `LongAdder` — highest throughput under contention; (2) `AtomicLong.incrementAndGet()` — atomic and gives a precise current value; (3) `synchronized` block — correct but adds lock overhead. Avoid `volatile int` for counters.

**Q: Given this code, what does it print?**
```java
AtomicInteger a = new AtomicInteger(5);
AtomicInteger b = new AtomicInteger(5);
System.out.println(a == b);
System.out.println(a.equals(b));
System.out.println(a.get() == b.get());
```
> **A:**
> ```
> false
> false
> true
> ```
> `a == b` is `false` — two distinct heap objects. `a.equals(b)` is also `false` — `AtomicInteger` inherits `Object.equals()`, which compares references, not values. `a.get() == b.get()` compares the unwrapped `int` primitives → `true`. This is a classic trap: always unwrap with `.get()` before comparing atomic values.

**Q: A framework needs to instantiate classes whose names are read from a config file and call a `process()` method on each, without knowing the classes at compile time. How do you implement this?**
```java
String className = config.get("handler"); // e.g. "com.app.OrderHandler"
Class<?> clazz = Class.forName(className);
Object instance = clazz.getDeclaredConstructor().newInstance();
Method process = clazz.getMethod("process");
process.invoke(instance);
```
> **A:** This is the classic reflective plugin/factory pattern. `Class.forName()` loads the class dynamically. `getDeclaredConstructor().newInstance()` replaces the deprecated `clazz.newInstance()` (which swallowed checked exceptions). `getMethod("process")` looks up the public method. Cache the `Method` object outside the loop — lookup is expensive; invocation is relatively cheaper. A better alternative where possible: define an interface `Handler` and have all classes implement it, then only use reflection to instantiate; cast to the interface for invocation to get JIT-inlinable direct calls.

**Q: You need to copy all field values from one object to another of the same type (a deep-ish clone) without a copy constructor. How do you implement this with reflection?**
```java
static <T> void copyFields(T source, T target) throws Exception {
    Class<?> clazz = source.getClass();
    while (clazz != null) {
        for (Field field : clazz.getDeclaredFields()) {
            if (Modifier.isStatic(field.getModifiers())) continue;
            field.setAccessible(true);
            field.set(target, field.get(source));
        }
        clazz = clazz.getSuperclass(); // walk up the hierarchy
    }
}
```
> **A:** Key points: (1) Use `getDeclaredFields()` — not `getFields()` — to include private fields. (2) Walk the superclass chain with `getSuperclass()` to capture inherited private fields. (3) Skip `static` fields — they belong to the class, not the instance. (4) `setAccessible(true)` is needed for private fields (watch for `InaccessibleObjectException` in Java 9+ modules). This only copies references for object fields (shallow copy for those); primitives are copied by value.

**Q: You are implementing a lock-free stack using `AtomicReference`. Why can the ABA problem occur here, and how would you fix it?**
```java
// Broken lock-free push:
AtomicReference<Node> top = new AtomicReference<>();

void push(Node newNode) {
    Node current;
    do {
        current = top.get();
        newNode.next = current;
    } while (!top.compareAndSet(current, newNode));
}
```
> **A:** Suppose Thread 1 reads `top = A`, Thread 2 pops A, pushes B, then re-pushes A (A is recycled). Thread 1's CAS sees `top == A` and succeeds — but A's `next` pointer may now point to freed/reused memory, corrupting the stack. Fix: use `AtomicStampedReference<Node>` — pair each node reference with a version stamp. The CAS checks both the reference and the stamp, so a recycled node with the same reference but a different stamp correctly fails:
> ```java
> AtomicStampedReference<Node> top = new AtomicStampedReference<>(null, 0);
> int[] stamp = new int[1];
> Node current = top.get(stamp);
> top.compareAndSet(current, newNode, stamp[0], stamp[0] + 1);
> ```

| `updateAndGet(fn)` | Atomic apply function, return new value (Java 8+) |
